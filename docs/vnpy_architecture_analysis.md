## 🎯 vn.py 事件驱动架构解析

### 核心组件

#### 1. Event 类
```python
class Event:
    def __init__(self, type: str, data: Any = None) -> None:
        self.type: str = type      # 事件类型标识
        self.data: Any = data      # 事件携带的数据
```

**特点**:
- 简单的事件封装，包含类型和数据
- 类型字符串用于事件分发
- 数据可以是任意 Python 对象

#### 2. EventEngine 类

**核心属性**:
| 属性 | 类型 | 说明 |
|------|------|------|
| `_interval` | int | 定时器间隔 (默认 1 秒) |
| `_queue` | Queue | 事件队列 (线程安全) |
| `_active` | bool | 引擎运行状态 |
| `_thread` | Thread | 事件处理线程 |
| `_timer` | Thread | 定时器线程 |
| `_handlers` | defaultdict | 事件类型→处理器列表 |
| `_general_handlers` | list | 通用处理器 (监听所有事件) |

**核心方法**:

| 方法 | 功能 |
|------|------|
| `start()` | 启动事件引擎 (事件处理线程 + 定时器线程) |
| `stop()` | 停止事件引擎 |
| `put(event)` | 将事件放入队列 |
| `register(type, handler)` | 注册特定事件的处理器 |
| `unregister(type, handler)` | 注销特定事件的处理器 |
| `register_general(handler)` | 注册通用处理器 |
| `unregister_general(handler)` | 注销通用处理器 |

**工作流程**:

```
┌─────────────────────────────────────────────────────────────┐
│                      EventEngine                             │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │ Timer Thread │───▶│  Event Queue │───▶│Event Handler │  │
│  │  (sleep 1s)  │    │   (Queue)    │    │   (Thread)   │  │
│  └──────────────┘    └──────────────┘    └──────────────┘  │
│         │                   │                    │          │
│         │  put(EVENT_TIMER) │  get(event)        │          │
│         │──────────────────▶│───────────────────▶│          │
│         │                   │                    │          │
│         │                   │             ┌──────┴──────┐   │
│         │                   │             │  Handlers   │   │
│         │                   │             │ - Specific  │   │
│         │                   │             │ - General   │   │
│         │                   │             └─────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 事件类型示例 (vn.py)

```python
EVENT_TIMER = "eTimer"              # 定时器事件
EVENT_TICK = "eTick"                # Tick 数据更新
EVENT_ORDER = "eOrder"              # 委托更新
EVENT_TRADE = "eTrade"              # 成交更新
EVENT_POSITION = "ePosition"        # 持仓更新
EVENT_ACCOUNT = "eAccount"          # 账户更新
EVENT_LOG = "eLog"                  # 日志事件
```

### 设计优势

1. **解耦**: 事件生产者和消费者完全解耦
2. **异步**: 事件处理在独立线程中异步执行
3. **可扩展**: 通过注册机制轻松添加新事件类型
4. **线程安全**: 使用 Queue 保证多线程安全
5. **灵活**: 支持特定事件处理器和通用处理器

## 📊 bobquant 当前架构分析

### 核心架构

**轮询模式**:
```python
def main_loop():
    while True:
        if s.is_trading_hours():
            try:
                run_check()          # 执行交易检查
                portfolio_summary()  # 组合汇总
            except Exception as e:
                _log(f"❌ 错误：{e}")
        else:
            _log("⏸️ 非交易时段，等待...")
        
        time.sleep(s.check_interval)  # 固定间隔 (10 秒)
```

### 三阶段处理流程

```
┌─────────────────────────────────────────────────────────────┐
│                      run_check()                             │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ Phase 1: 网格做 T                                     │   │
│  │ - 并行获取持仓股实时价格 (ThreadPoolExecutor)         │   │
│  │ - 检查止盈/止损 (risk.check)                          │   │
│  │ - 高抛操作 (grid_t.check_sell)                        │   │
│  │ - 接回操作 (grid_t.check_buyback)                     │   │
│  │ - 兜底接回 (已清仓但未接回的股票)                      │   │
│  └──────────────────────────────────────────────────────┘   │
│                            │                                 │
│                            ▼                                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ Phase 2: 风控                                         │   │
│  │ - 止盈检查 (take_profit)                              │   │
│  │ - 止损检查 (stop_loss)                                │   │
│  │ - 移动止损 (trailing_stop)                            │   │
│  └──────────────────────────────────────────────────────┘   │
│                            │                                 │
│                            ▼                                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ Phase 2.5: 情绪指数主动仓位管理                        │   │
│  │ - 计算当前仓位                                        │   │
│  │ - 检查是否需要减仓                                    │   │
│  │ - 执行减仓操作                                        │   │
│  └──────────────────────────────────────────────────────┘   │
│                            │                                 │
│                            ▼                                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ Phase 3: 策略信号                                     │   │
│  │ - 获取历史数据 (ML 配置天数)                           │   │
│  │ - 获取传统技术指标信号 (MACD, Bollinger 等)            │   │
│  │ - 综合决策引擎 (ML + 情绪 + TA)                        │   │
│  │ - 执行买入/卖出操作                                   │   │
│  └──────────────────────────────────────────────────────┘   │
│                            │                                 │
│                            ▼                                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 保存：account.save() + executor.sync_trade_log()     │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 数据流

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ DataProvider │───▶│  Strategy    │───▶│  Executor    │
│ (并行获取)    │    │  Engine      │    │  (执行交易)  │
└──────────────┘    └──────────────┘    └──────────────┘
       │                    │                    │
       │                    │                    │
       ▼                    ▼                    ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ Tencent API  │    │ ML Predictor │    │ Account      │
│ Baostock     │    │ Sentiment    │    │ Broker API   │
└──────────────┘    └──────────────┘    └──────────────┘
```

### 当前架构特点

| 特点 | 描述 |
|------|------|
| **同步轮询** | 固定 10 秒间隔检查所有股票 |
| **并行数据获取** | ThreadPoolExecutor 并行获取行情 (26 只股票从 2.5 秒降至 0.3 秒) |
| **三阶段处理** | 做 T → 风控 → 策略信号，顺序执行 |
| **状态管理** | GridTStrategy 维护做 T 状态 |
| **综合决策** | DecisionEngine 整合 ML、情绪、技术指标 |

### 存在的问题

1. **固定间隔延迟**: 10 秒轮询间隔可能导致信号延迟
2. **资源浪费**: 即使没有事件也定时检查所有股票
3. **响应性差**: 紧急风控事件需要等待下一个轮询周期
4. **扩展性受限**: 添加新事件类型需要修改主循环
5. **阻塞风险**: 单个股票处理慢会影响整体检查

| 维度 | vn.py 事件驱动 | bobquant 轮询 |
|------|--------------|--------------|
| **响应性** | ⭐⭐⭐⭐⭐ 即时响应事件 | ⭐⭐⭐ 最多延迟 10 秒 |
| **资源效率** | ⭐⭐⭐⭐ 仅在事件发生时处理 | ⭐⭐ 固定间隔检查所有股票 |
| **解耦性** | ⭐⭐⭐⭐⭐ 完全解耦 | ⭐⭐⭐ 模块间耦合较高 |
| **可扩展性** | ⭐⭐⭐⭐⭐ 注册新处理器即可 | ⭐⭐⭐ 需修改主循环 |
| **实现复杂度** | ⭐⭐⭐ 中等 (需理解异步) | ⭐⭐⭐⭐ 简单直观 |
| **调试难度** | ⭐⭐ 异步调试较复杂 | ⭐⭐⭐⭐ 同步易于追踪 |
| **线程安全** | ⭐⭐⭐⭐ Queue 保证安全 | ⭐⭐⭐ 单线程主循环 |
| **适用场景** | 高频、多事件源、实时性要求高 | 低频、批量处理、简单场景 |

## 💡 改进建议

### 方案 A: 完全事件驱动重构 (推荐长期目标)

**架构设计**:
```python
# 事件类型定义
EVENT_TICK_UPDATE = "eTickUpdate"        # Tick 数据更新
EVENT_SIGNAL_GENERATED = "eSignal"       # 信号生成
EVENT_ORDER_SUBMITTED = "eOrder"         # 委托提交
EVENT_TRADE_EXECUTED = "eTrade"          # 成交执行
EVENT_RISK_TRIGGERED = "eRisk"           # 风控触发
EVENT_TIMER_10S = "eTimer10s"            # 10 秒定时器
EVENT_TIMER_1MIN = "eTimer1min"          # 1 分钟定时器
EVENT_MARKET_OPEN = "eMarketOpen"        # 开盘事件
EVENT_MARKET_CLOSE = "eMarketClose"      # 收盘事件
```

**核心组件**:
```
┌─────────────────────────────────────────────────────────────┐
│                      EventEngine                            │
│  (基于 vn.py EventEngine 扩展)                              │
└─────────────────────────────────────────────────────────────┘
         │
         │ put(event)
         ▼
┌─────────────────────────────────────────────────────────────┐
│  Event Queue (Thread-Safe)                                  │
└─────────────────────────────────────────────────────────────┘
         │
         │ get(event)
         ▼
┌─────────────────────────────────────────────────────────────┐
│  Event Handlers                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │ TickHandler │  │SignalHandler│  │RiskHandler  │         │
│  │ (数据更新)   │  │ (信号生成)  │  │ (风控执行)  │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │OrderHandler │  │TradeHandler │  │TimerHandler │         │
│  │ (委托管理)  │  │ (成交处理)  │  │ (定时任务)  │         │
│  └─────────────┘  └─────────────┘  └─────────────┘         │
└─────────────────────────────────────────────────────────────┘
```

**优势**:
- ✅ 实时响应市场变化
- ✅ 模块完全解耦
- ✅ 易于添加新功能
- ✅ 支持并发处理

**劣势**:
- ❌ 重构成本高
- ❌ 异步调试复杂
- ❌ 需要重新设计数据流

### 方案 B: 混合架构 (推荐中期过渡)

**设计思路**: 保留轮询主循环，引入事件机制处理特定场景

```python
# 保留现有轮询框架
def main_loop():
    while True:
        if s.is_trading_hours():
            run_check()  # 保留现有逻辑
        time.sleep(s.check_interval)

# 新增事件引擎处理异步事件
class HybridEventEngine:
    def __init__(self):
        self.engine = EventEngine(interval=1)
        self.register_handlers()
    
    def register_handlers(self):
        # 注册风控事件处理器 (紧急事件)
        self.engine.register(EVENT_RISK_TRIGGERED, self.on_risk_event)
        # 注册通知事件处理器
        self.engine.register(EVENT_NOTIFY, self.on_notify_event)
    
    def on_risk_event(self, event):
        # 立即执行风控操作，不等待轮询
        code, action, shares = event.data
        execute_risk_action(code, action, shares)
    
    def on_notify_event(self, event):
        # 异步发送通知，不阻塞主循环
        title, message = event.data
        send_feishu_async(title, message)
    
    def start(self):
        self.engine.start()
    
    def trigger_risk_event(self, code, action, shares):
        # 外部可调用，触发风控事件
        event = Event(EVENT_RISK_TRIGGERED, (code, action, shares))
        self.engine.put(event)
```

**适用场景**:
- 紧急风控事件 (止损、止盈)
- 异步通知发送
- 日志记录
- 后台数据同步

### 方案 C: 优化轮询 (推荐短期实施)

**优化点**:

1. **动态调整轮询间隔**:
```python
# 根据市场活跃度动态调整
if is_high_volatility():
    check_interval = 5  # 高波动时 5 秒
else:
    check_interval = 10  # 正常时 10 秒
```

2. **优先级队列**:
```python
# 优先处理持仓股和信号股
priority_codes = get_priority_codes()  # 持仓 + 信号股
normal_codes = get_normal_codes()      # 其他观察股

# 先处理优先级高的
run_check_for_codes(priority_codes)
# 时间允许再处理其他
if time_remaining > 0:
    run_check_for_codes(normal_codes)
```

3. **增量检查**:
```python
# 只检查价格变化超过阈值的股票
changed_codes = [code for code in codes 
                 if abs(price_change[code]) > threshold]
run_check_for_codes(changed_codes)
```
